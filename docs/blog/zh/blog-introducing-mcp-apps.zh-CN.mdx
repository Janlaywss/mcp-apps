# Module Federation MCP Apps - 用 Module Federation 驱动 MCP Apps UI

AI 对话窗口里可以跑一个完整的应用了。

[MCP Apps](https://modelcontextprotocol.io/docs/extensions/apps) 标准让 Claude Desktop、VS Code、Goose 等 AI 宿主能在对话中**直接渲染可交互的 UI**——部署向导、故障工单、数据仪表盘，全部内嵌在聊天窗口里，操作结果实时回传给 AI。

但如果你的团队已经有一套成熟的前端体系，想把现有组件接入 MCP，现实并不像官方示例那么简单。

本文介绍 **[`@module-federation/mcp-apps`](https://github.com/module-federation/mcp-apps)**——用 Module Federation 解决 MCP Apps 的工程化落地问题，让任何规模的项目都能以极低成本把现有 UI 接入 MCP，而不需要为此重建一套构建和部署体系。

> ⭐ 如果觉得有用，欢迎给项目点个 Star：**[module-federation/mcp-apps](https://github.com/module-federation/mcp-apps)**

## MCP Apps 是什么？

传统 MCP 工具的模式是：工具调用 → 返回文本 → AI 处理 → 输出文字。这套模式可行，但在以下场景会遇到根本性障碍：

- **结果不可操作**：AI 建议"重启 nginx 实例"，但用户还是要自己去控制台操作
- **多步骤任务低效**：部署一个服务需要选环境、填配置、确认参数……靠对话来回确认，上下文随时可能断掉
- **上下文窗口是有限资源**：挂 10 个服务 × 每个 5 个工具，工具描述本身就烧掉几千 Token，AI 还没干活

MCP Apps 是 [MCP 协议](https://modelcontextprotocol.io/)的官方扩展规范，由 Anthropic 联合 OpenAI、Block、VS Code、JetBrains、AWS 共同制定，目前在 Claude、Claude Desktop、VS Code GitHub Copilot、Goose 等客户端中支持。

**核心思路**：给工具加上一个 UI 层。工具不只是"函数 + 返回值"，工具可以**声明一个 `ui://` 资源 URI**。当 LLM 决定调用这个工具时：

```
LLM 决定调用工具
    │
    ▼
宿主获取 ui:// 资源 ──→ 服务端返回打包 HTML
    │
    ▼
在对话里渲染沙箱 iframe ──→ 用户看到可交互界面
    │                          （表单、向导、仪表盘……）
    ▼
用户操作 ──→ 结果通过 postMessage 回传给 AI ──→ 对话继续推进
```

## 什么时候该用 MCP Apps

**探索复杂数据**
用户问"按地区显示销售额"。文字回应只能列出数字，而 MCP App 可以渲染一张可交互的地图——用户点击区域下钻、悬停查看详情、在不同指标之间切换，无需追加任何提示词。

**需要配置多个选项**
设置一个部署环境涉及数十个相互依赖的选项。与其来回对话（"哪个区域？""实例规格？""开启自动扩缩容？"），MCP App 可以直接呈现一个表单，用户一次看到所有选项，还有校验和默认值。

**展示富媒体内容**
当用户想要查阅 PDF、查看 3D 模型或预览生成的图片时，文字描述远远不够。MCP App 可以在对话里直接嵌入真正的查看器（平移、缩放、旋转）。

**实时监控**
显示实时指标、日志或系统状态的仪表盘需要持续更新。MCP App 维护一个持久连接，数据变化时自动更新显示，无需用户反复问"现在状态怎么样？"。

**多步骤工作流**
审批报销单、审查代码变更或对问题进行分类，需要逐一检查每个条目。MCP App 可以提供导航控件、操作按钮，以及在多次交互之间持久保存的状态。


**安全隔离**：MCP App UI 运行在宿主控制的沙箱 iframe 中，无法访问父页面 DOM、Cookie 或 localStorage，宿主无需完全信任服务端作者就能安全渲染第三方界面。所有通信通过 postMessage 进行。

## 问题：UI 从哪里来？

MCP Apps 规范定义了一套清晰的协议：工具被调用时，服务端可以返回一个 `ui://` 资源 URI，宿主获取它、渲染一个沙箱 iframe，用户就能在对话中看到可交互的组件。

**官方推荐的做法**是自己写一个完整的 MCP Server：注册工具和资源、用 Vite 把 UI 打包成自包含 HTML、通过 Express 提供服务。这套方案完整且可控，如果你在从零搭建一个独立的 MCP App，这正是你需要的——官方的[构建指南](./official-build-guide.md)详细讲解了这整个流程。

但如果你的团队已经有一套成熟的前端体系，这套"从零搭建"的方案在实际操作中根本跑不通。

现实中的业务应用不是一个 `get-time` 组件。它们是有几十上百个页面的大型应用，组件之间共享状态、共享 Design System、共享数据请求层，深度依赖项目内部的 context 和基础设施。你想把其中一个 `DeployWizard` 抽出来单独打包，面对的是：

- 它依赖 `src/store`，而 store 依赖 `src/api`，而 api 依赖认证模块……
- 它用了公司内部的 UI 组件库，这个库没有独立发布，只在 monorepo 里
- 它和另外三个组件共享一个全局的 `AppContext`，抽出来之后 context 断了
- 你有 30 个需要暴露的工具，就需要维护 30 套独立的 `vite.config`、30 个 bundle、和一条与主流水线完全并行的"影子 CI/CD"

更根本的矛盾是：**你永远在追赶主应用的变更**。主应用每次迭代，你都要把改动同步到 MCP Server 这边的 bundle。组件改了一个 props，MCP 这边要重新抽取、重新构建、重新部署。两套流水线，两套部署节奏，两套可能不一致的状态。

规范没有说**你该怎么解决这个同步问题**——而最直接的做法只是把 UI 打包进 MCP Server，把问题保留下来：

```
┌──────────────────────┐          ┌──────────────────────────┐
│  你的主应用           │  每次改动  │  MCP Server bundle       │
│                      │  ───────▶ │                          │
│  DeployWizard v2.3   │  需手动追  │  deploy-wizard.html v?   │  ← 谁知道和主应用对不对齐
│  IncidentForm v1.8   │  赶同步   │  incident-form.html v?   │
│  UserLookup v3.1     │           │  user-lookup.html v?     │
│  ... 30 个组件       │           │  ... 30 个 bundle        │
└──────────────────────┘          └──────────────────────────┘
       主 CI/CD 流水线                   影子 CI/CD 流水线
```

这是一个对大型团队来说根本无法长期维护的结构。

## 解决方案：让 UI 留在原地

根本问题在于：官方方案把 UI **打包进** MCP Server——构建时捆绑，部署时捆绑，于是 UI 的每一次变更都必须触发 Server 重新构建和部署。`@module-federation/mcp-apps` 反转了这个绑定关系：**UI 不再进入 Server，Server 只持有一个指向 UI 的地址。**

### 三个核心文件

**`mcp_apps.json`（唯一配置入口）**
这是你唯一需要维护的文件。它告诉 Server：有哪些远程、每个工具对应哪个组件、CDN 地址是什么。新增工具 = 加一行 JSON；更新 UI = 改版本号字符串。不需要代码改动，不需要重新编译 Server。

**`mcp-app.html`（壳页面，内置于 Server）**
Server 提供的 HTML 只有不到 1 KB——它不包含任何业务 UI，只是一个 Module Federation 的动态加载容器。宿主每次请求资源时，Server 返回这个壳；壳在运行时从 CDN 按需拉取你的真实组件。

**你的前端部署产物（按原有节奏发布）**
组件继续通过你现有的 CI/CD 构建、发布到 CDN，完全不知道 MCP 的存在。Server 端的 `mcp_apps.json` 里的版本号就是两者之间的唯一"接缝"——你决定什么时候让 MCP 切到新版本。

### 机制：运行时加载 vs 构建时打包

**标准做法（UI 打包进 Server）：**
- UI 在**构建时**打包进 Server，和 Server 代码捆绑在一起
- bundle 存在 Server 本地，UI 改动 = Server 重建 = Server 重新部署
- 每个 bundle 独立打包全量依赖，10 个工具 × 组件体积 = 冷启动代价
- 依赖无法跨工具共享，React、Design System 每个 bundle 各打一份

**`@module-federation/mcp-apps`（UI 留在 CDN）：**
- UI **永远不进入** Server，Server 只持有一个指向 CDN 的版本号字符串
- UI 改动不触发 Server 任何变更，前端按自己节奏发布，MCP 侧只改版本号
- 冷启动只需下发 **~1 KB 壳**，组件按需从 CDN 懒加载
- MF shared scope 跨工具共享依赖——React、公共库全局只加载一次

因为 UI 是**运行时**通过 Module Federation 从 CDN 加载的，MCP Server 对 UI 的内容和版本变更完全无感知——你的前端团队按自己的节奏发布，MCP 侧只在你主动更新版本号时才会切换。这就是"影子 CI/CD"问题消失的原因：**根本不存在第二条流水线**。

```
mcp_apps.json
     │
     │  "从 https://cdn.example.com/my-app@1.2.3
     │   加载 ./DeployWizard"
     │
     ▼
┌─────────────────────┐  ① 工具注册   ┌─────────────────┐
│  MF MCP Server      │─────────────▶│  AI 宿主         │
│  （零 UI bundle）    │  _meta.ui     │  (Claude 等)    │
│                     │◀─────────────│                 │
│  提供 mcp-app.html  │  ② 读取资源   └────────┬────────┘
└─────────────────────┘                       │ ③ 渲染 iframe
                                       ┌──────▼───────────────┐
                                       │  mcp-app.html        │
                                       │  加载 MF 远程         │
                                       │  渲染组件             │
                                       └──────────┬───────────┘
                                                  │ ④ 已在 CDN 上
                                             ┌────┴─────┐
                                             │  你的前端 │
                                             │  部署产物 │
                                             └──────────┘
```

## 工作原理

对应上方架构图的四个步骤：

**① 工具注册**：Server 启动时读取 `mcp_apps.json`，为每个工具调用 `registerAppTool`，在工具声明的 `_meta.ui.resourceUri` 字段中写入对应远程的 `ui://` URI（每个 remote 对应一个唯一 URI）。宿主在 MCP 握手阶段拿到完整的工具列表和 UI 能力声明。

**② 宿主读取资源**：用户触发工具调用后，宿主识别到 `_meta.ui`，主动向 Server 发起资源读取请求（`resources/read`）。Server 收到请求，返回内置的 `mcp-app.html` 壳页面（约 1 KB），同时携带该 remote 的 CSP 配置作为 `_meta`。壳页面本身**不含任何业务 UI 代码**。

**③ 宿主渲染 iframe**：宿主将壳页面内容注入一个沙箱 iframe，在对话窗口中展示。工具调用的入参（`args`）通过 `postMessage` 注入 iframe，作为组件的初始 props。

**④ 壳页面从 CDN 加载 MF 远程**：`mcp-app.html` 在浏览器端运行，根据工具入参中的 `moduleFederation` 配置（`remoteEntry`、`module`、`exportName`）动态拉取你 CDN 上的 `remoteEntry.js`，解析 Module Federation 的 shared scope，按需加载目标组件并渲染。整个过程发生在运行时，Server 端对组件内容完全无感知。

**双向通信**：组件通过 `mcpApp.sendMessage` 向宿主 postMessage，宿主将消息注入对话上下文，AI 继续推进任务或触发下一个工具调用。

## 如何接入

### 第一步：把你的前端项目接入 Module Federation

`@module-federation/mcp-apps` 的核心机制是在运行时从 CDN 动态加载你的组件，这依赖 Module Federation 的远程模块能力。如果你的项目还没有接入 MF，可以参考官方快速上手文档完成配置：

👉 [Module Federation 快速上手（Webpack / Rspack / Vite）](https://module-federation.io/zh/guide/start/quick-start.html)

配置完成后，在 `exposes` 里声明你想暴露给 MCP 的组件，构建产物发布到 CDN 即可。已有 MF 远程的项目**跳过此步**。

### 第二步：提供 `mcp_apps.json` 注册文件

> 已有 MF 接入的团队，这就是你的实际起点。

`mcp_apps.json` 是整个方案的唯一配置入口，也是 Server 和你的前端部署之间唯一的"接缝"。它做两件事：

- **声明 remotes**：告诉 Server 你的 MF 远程在哪里——CDN 地址、版本号、CSP 域名白名单
- **声明 tools**：将每个 MF `exposes` 条目映射为一个 MCP 工具——工具名、描述（展示给 AI）、对应的 remote 和模块路径

Server 启动时读取这个文件完成全部注册，之后**不需要任何代码改动**：新增工具加一行 JSON，更新 UI 改版本号字符串，重启 Server 即生效。

```json title="mcp_apps.json"
{
  "remotes": [
    {
      "name": "@my-org/ops-tools",
      "version": "2.4.1",
      "baseUrl": "https://cdn.example.com/ops-tools/2.4.1"
    }
  ],
  "tools": [
    {
      "name": "deploy_wizard",
      "title": "部署向导",
      "description": "三步交互式部署 UI",
      "remote": "@my-org/ops-tools",
      "module": "./DeployWizard"
    }
  ]
}
```

> **完整字段说明**见 [mcp_apps.json 完全指南](../zh/mcp_apps_json.md)——涵盖每个字段的语义、约束条件、CSP 配置规则，以及与 `module-federation.config.ts` 的完整对应关系。

### 第三步：启动 MF MCP Server

`@module-federation/mcp-apps` 内置了一个**完整实现 [MCP Apps 标准](https://modelcontextprotocol.io/docs/extensions/apps)的 MCP Server**——支持工具注册、`ui://` 资源 URI 生成、iframe 壳页面服务等全链路能力，无需你自己实现任何 MCP 协议。

Server 的启动命令是一条 `npx`：

```bash
npx -y @module-federation/mcp-apps@latest \
  --config /绝对路径/mcp_apps.json \
  --stdio
```

**以 Claude Desktop 为例**，将以下配置写入 `~/Library/Application Support/Claude/claude_desktop_config.json`：

```json
{
  "mcpServers": {
    "my-mcp-app": {
      "command": "npx",
      "args": [
        "-y",
        "@module-federation/mcp-apps@latest",
        "--config",
        "/绝对路径/mcp_apps.json",
        "--stdio"
      ]
    }
  }
}
```

重启 Claude Desktop，工具列表中就会出现你在 `mcp_apps.json` 里声明的所有工具。

> **nvm 用户注意**：Claude Desktop 不继承 shell 的 PATH，`npx` 可能解析到错误的 Node 版本。建议使用 `which npx` 获取完整路径并通过 `env` 固定 PATH：
> ```json
> {
>   "mcpServers": {
>     "my-mcp-app": {
>       "command": "/Users/you/.nvm/versions/node/v22.21.1/bin/npx",
>       "args": ["-y", "@module-federation/mcp-apps@latest", "--config", "/绝对路径/mcp_apps.json", "--stdio"],
>       "env": {
>         "PATH": "/Users/you/.nvm/versions/node/v22.21.1/bin:/usr/local/bin:/usr/bin:/bin"
>       }
>     }
>   }
> }
> ```

其他支持 MCP 的客户端（VS Code GitHub Copilot、Goose、Cursor 等）配置方式类似，均填写相同的 `command` 和 `args`。

## 与 AI 通信

接入完成后，你的组件已经能在 AI 宿主中渲染。如果你希望组件在用户操作后**主动推进对话**——比如表单提交后告诉 AI 下一步该做什么——可以通过 `mcpApp` prop 实现。

`mcpApp` 由框架在渲染时**自动注入为 prop**，不需要安装任何额外依赖，也不需要任何 import，在组件 props 里声明接口即可使用：

```tsx
interface McpApp {
  sendMessage?: (params: {
    role: string;
    content: Array<{ type: string; text: string }>;
  }) => Promise<{ isError?: boolean }>;
}

export default function IncidentForm({ mcpApp }: { mcpApp?: McpApp }) {
  const handleSubmit = async (incident) => {
    // 正常的业务逻辑
    await fetch('/api/incidents', { method: 'POST', body: JSON.stringify(incident) });

    // 告诉 AI 发生了什么，让对话继续
    await mcpApp?.sendMessage({
      role: 'user',
      content: [{ type: 'text', text: `故障单 #${incident.id} 已创建，严重级别：${incident.severity}` }]
    });
  };

  return <IncidentFormUI onSubmit={handleSubmit} />;
}
```

`mcpApp` 在组件运行于 MCP 宿主之外时为 `undefined`，现有 UI 在任何场景下都能正常工作，**零侵入**。不想接入双向通信的组件完全不需要改动。

## 接入你自己的项目

MF 远程 `exposes` 配置里的任意组件都可以成为 MCP 工具。如果你的 `module-federation.config.ts` 里已有：

```ts
exposes: {
  './IncidentForm': './src/components/IncidentForm',
  './UserLookup': './src/components/UserLookup',
}
```

你已经有两个 MCP 工具 UI 了，一行代码都不用改。

### 用 AI 生成 `mcp_apps.json`

仓库内置了一个 AI Skill，能自动检测你的 MF 配置并生成 JSON。把 [`skill/generate-mcp-apps-config.md`](https://github.com/module-federation/mcp-apps/blob/main/skill/generate-mcp-apps-config.md) 丢给任意 AI 编程工具并说：

```
请按照 skill/generate-mcp-apps-config.md 的指引为我的项目生成 mcp_apps.json
```

AI 会读取你的 `module-federation.config.ts`，识别包名和 exposes，询问你的 CDN 地址和版本号，然后输出一份可以直接使用的配置。生成后按[如何接入](#如何接入)的步骤启动 Server 即可。

> 想手动写？完整字段说明见 [mcp_apps.json 完全指南](../zh/mcp_apps_json.md)。

## HTTP 模式：支持自定义 Agent 框架

Claude Desktop 使用 stdio。但如果你在构建自定义 Agent 框架——MCP 客户端运行在服务端、iframe 在浏览器里渲染——Server 也支持 HTTP 模式：

```bash
npx -y @module-federation/mcp-apps@latest \
  --config /path/to/mcp_apps.json \
  --http \
  --port 3001
```

HTTP Server 暴露以下端点：
- `POST /mcp-rpc` — 供 `@byted/mcp-client-sdk` 使用的 JSON-RPC 端点
- `GET /static/mcp-app-shell.html` — iframe 壳页面
- `GET /proxy` — 浏览器跨域请求的 CORS 代理

## 常见问题

**我的组件需要改写吗？**

不需要。MF 远程里现有的任意 React 组件都可以直接用。`mcpApp` prop 是可选的（框架不注入时为 `undefined`），现有行为完全不受影响。

**如果 AI 宿主不支持 MCP Apps 怎么办？**

Server 在 MCP 握手阶段会检测宿主的能力。若宿主不支持 MCP Apps 规范，工具会返回纯文本响应而不是 `ui://` URI。工具依然可用，只是用户看不到交互 UI。

**不用 Module Federation 能接入吗？**

处理动态远程加载的 `mf_render_container` 壳内部使用了 Module Federation，但你的组件只需要能以标准 MF 远程的方式对外提供服务即可。如果从头开始搭，MF 的配置成本很小——参考 [module-federation.io](https://module-federation.io) 的 Webpack、Rspack、Vite 快速入门。

**UI 有沙箱隔离吗？**

有。组件运行在 AI 宿主内的沙箱 iframe 中。CSP 通过 `mcp_apps.json` 里的 `csp` 字段强制执行，控制组件可以访问哪些域名。

**鉴权怎么处理？**

组件运行在用户的浏览器上下文（或宿主的 webview）中，因此可以使用 cookie、token 或你前端已有的任何鉴权机制。MCP Server 本身是无状态的——它只负责生成 iframe HTML。
